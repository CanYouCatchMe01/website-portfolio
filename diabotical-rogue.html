DirectX12 port:

Porting the Glitch Engine to DirectX12 has been the most fun I’ve had so far at The GD Studio. The task is still in progress, with about 80% of the rendering code successfully transitioned from DirectX11 to DirectX12. I’ve always found rendering programming fascinating, though it was a bit challenging to pick up initially. This porting process has given me a much deeper understanding of graphics programming, especially around performance bottlenecks and effective memory management. Now that I’ve gotten hands-on experience with DirectX12, I feel much more confident, and I’m sure picking up other graphics APIs in the future will be significantly easier.

Level editor:

The level editor allows mappers to place and arrange objects to create visually compelling and gameplay-rich maps. I worked closely with the mapping team, fixing bugs, iterating on feedback, and implementing new features based on their needs.

Occlusion culling and triangle remover:

To optimize performance, it's important to reduce the number of triangles rendered on screen. One of the most straightforward techniques is frustum culling, which improves efficiency by rendering only objects within the camera's view. If an object is outside the camera's frustum, meaning it's behind the player or off to the side, it gets skipped during rendering, saving processing power.

While frustum culling improves performance by skipping objects outside the camera's view, it doesn’t account for occlusion when one object blocks another from view. For example, if a large sphere is in front of several smaller spheres, frustum culling would still render all of them, even though the smaller ones are completely hidden. This is where occlusion culling comes in, allowing us to avoid rendering objects that are obscured by others.

In Diabotical Rogue, we use a baked occlusion culling system optimized for static geometry. Since the levels contain many non-moving objects, this method allows us to precompute visibility and significantly boost runtime performance.

The process begins with level designers placing visibility volumes throughout the map, these define areas from which visibility will be calculated. They then place bake points inside these volumes, which serve as virtual camera positions during the baking process.

When the bake process is triggered, the system moves the camera to each bake point and uses DirectX occlusion queries to determine which objects are visible from that point within each volume. The results are saved into a visible object list for each volume.

At runtime, when the player is within a given volume, the engine references that volume’s visible object list to decide what to render. If an object isn’t in the list, it’s not drawn, saving on rendering cost and improving performance. Because visibility is precomputed, the game can perform fast and efficient culling without real-time occlusion checks.

Triangle remover:

Many triangles in the game are never visible from any direction so those triangles we can remove. In Rogue, we for example remove 55% of the triangles in the map Wellspring. The triangles that can be removed are for example triangles that are inside of other objects. Like for example here with the two rocks that intersect each other, we can remove the intersection triangles because they are never visible.

Before triangle remover:

After triangle remover:

The triangle remover scans for visible triangles while we bake the occlusion culling. During the baking process, each triangle is assigned a unique ID. All visible triangles draw their ID to a render target texture. At the end of the frame, we read every pixel of that texture to determine which triangles are visible. Any triangle that remains hidden throughout the baking process is removed.
